# **0018: Asynchronous Replay Upload Queue**

**Context:** The application automatically uploads user-saved .replay files to ballchasing.com. Because Node.js is single-threaded, directly executing and awaiting a heavy HTTP POST request inside the main event loop risks blocking the Engine for several seconds if the user's network drops or the third-party API hangs. Blocking the Engine will cause it to drop live 120Hz WebSocket telemetry ticks if the user has already queued into their next match.  
**Decision:** The Engine will utilize an Asynchronous Replay Upload Queue and a File System Watcher. The watcher detects manually saved replays in the TAGame/Demos folder and pushes the filepath to the queue. The queue processes uploads strictly in the background via non-blocking asynchronous promises (or a dedicated Worker Thread), ensuring the main telemetry loop remains untouched. The Engine will broadcast explicit UploadStatus WebSocket events (e.g., Pending, Success, Failed) to the Presentation Layer to trigger UI snackbar notifications.  
**Consequences:** Complete isolation of network I/O from telemetry ingestion. The overlay can safely inform the user of upload status without interrupting the Live Match State, perfectly handling the delayed nature of manual post-match saves.
